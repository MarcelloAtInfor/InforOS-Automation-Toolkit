DemoInvoiceLoader_V4 Delivery Report

Prepared on 2026-03-08 for the deterministic rewrite of the invoice automation flow.

Executive Summary

DemoInvoiceLoader_V4 replaces the prior GenAI-driven transactional path with an RPA-driven, deterministic CSI API workflow. Live validation in tenant SLSGDENA052_DEM showed that V4 can:

  • extract invoice data through live OCR,
  • reuse or create vendors deterministically,
  • reuse or create items deterministically,
  • create purchase orders and PO lines,
  • persist tax correctly to the visible PO tax field,
  • send end-user ION notifications, and
  • fail closed when vendor identity is unsafe.

What Changed

Area Old Project V4
Transaction logic GenAI chat/tool orchestration Direct deterministic CSI IDO calls from RPA
Vendor matching Underdetermined, sometimes reused the wrong vendor Exact-name gate with structured address scoring, bounded typo tolerance, fail-closed ambiguity handling
Item handling GenAI-driven search/create path Deterministic search/create/verify path with persisted UOM reuse
Tax Initially missing or misdirected Mapped to visible SalesTaxT and validated live
Error behavior Could drift or partially succeed opaquely Explicit status, verification, notifications, and fail-closed decisions

Final Workflow Shape

  • MainPage.xaml is the public batch root.
  • ProcessDocumentBatch.xaml builds the worklist, claims files, loops them, and writes batch artifacts.
  • SingleDocumentController.xaml performs OCR, intake formatting, vendor resolution, item resolution, PO creation, verification, and per-document notification.
  • Deploy-time tenant-specific values are materialized into .deploy\DemoInvoiceLoader_V4_<TENANT>.

Key Fixes Landed

  • Vendor matching now uses exact-name gating, normalized street/city/state/zip scoring, safe abbreviation normalization, bounded street typo tolerance, and ambiguity fail-closed behavior.
  • Vendor state is explicitly persisted and verified on creation.
  • Reused items now propagate the persisted CSI UOM into PO lines rather than trusting OCR UOM.
  • PO tax is written to the visible UI field SalesTaxT.
  • Notification payload now includes item codes and unhandled controller exceptions still route to notification.
  • The published project identity is now DemoInvoiceLoader_V4 with a unique RPA processId, so it will not overwrite the old V3 process.

Operational Notes

  • Tracked source remains repo-safe. Tenant-specific values are expected to be applied through prepare_deploy.py.
  • Live OCR still requires Studio rebind/verification in a newly generated .deploy copy because OCR activity metadata is partly Studio-managed.
  • The attended PowerShell runner is useful, but it is still focus-based desktop automation and not a true background API.

Suggested Next Steps

  1. Publish and validate one final tenant-side run from the newly published V4 process.
  2. Document the operator runbook: input folder usage, expected notifications, and ambiguity failure handling.
  3. Add a lightweight reconciliation/report export that summarizes created vendors/items/POs after each batch.
  4. If desired, add a controlled human-review queue for fail-closed vendor ambiguity cases rather than stopping at notification only.
  5. Optionally replace the focus-based Studio helper with a stronger Windows automation bridge if attended iteration remains common.

Recommendation

V4 is in a materially better state than the prior GenAI transactional flow for real use. The rewrite removed the largest source of nondeterminism without losing OCR, notifications, or end-to-end automation. The remaining work is operational polish, not core architecture repair.
