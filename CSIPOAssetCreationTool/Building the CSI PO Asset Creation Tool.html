The Development Process
How Claude Code Works with a Cloud Platform
Building on a proprietary cloud platform like Infor OS is fundamentally different from typical local development. There's no "npm install" or "pip install" that gives you a local stack. Every component lives in the cloud, accessed only through REST APIs, and the platform has its own domain-specific concepts, naming conventions, and quirks. Here's how the development loop worked.
1

Explore & Discover

Read Swagger specs, call APIs exploratively, discover property names via /info/{ido} endpoints, map out the domain model. Claude Code ingested API specs and built mental models of the platform.

2

Script & Deploy

Generate Python scripts to create tools/agents via REST API. Each tool definition includes api_docs that another LLM will interpret. Deploy to the Infor GenAI platform. No local runtime — everything runs in the cloud.

3

Test via Chat API

Send test prompts through the Chat Service API to invoke the deployed agent. Parse the LLM response. Check if the agent called the right tools with the right parameters. Debug 404s, malformed URLs, missing properties.

4

Iterate & Document

Fix issues, redeploy, re-test. Write fix scripts. Update CLAUDE.md with discoveries. Update log.md with progress. Each fix informs the next session. The knowledge base compounds.

Analysis
Strengths & Challenges
Using Claude Code to build on a proprietary cloud platform is a fundamentally different experience from local-stack development. These are the observed strengths and challenges from this project.

Strengths Where Claude Code Excelled

API Exploration & Script Generation

Claude Code ingested Swagger specs (3,000+ line JSON files) and immediately generated working API interaction scripts. It could reason about OAuth flows, IDO query patterns, and REST conventions to produce correct API calls on the first attempt in most cases.

XAML Workflow Creation from Scratch

RPA workflows were generated as raw XAML — a format that is extremely verbose, with strict requirements for ViewStateData, activity namespaces, and VBScript expressions. Claude Code produced valid, runnable XAML consistently, including complex patterns like nested TryCatch blocks, ForEach loops with JToken parsing, and async polling.

Systematic Debugging

When ContinueOnError was silently masking failures across 5 workflows, Claude Code performed a systematic audit of every activity, identified the pattern, and fixed each occurrence. This kind of exhaustive, methodical analysis across multiple files is where AI assistance truly shines.

Knowledge Accumulation via CLAUDE.md

The CLAUDE.md convention created a compounding knowledge base. Every API quirk (IDO case-sensitivity, VBScript vs .NET, api_docs whitespace) was documented once and available to all future sessions. This turned ephemeral debugging into permanent institutional knowledge.

Rapid Iteration Velocity

The cycle of write script → deploy to cloud → test via API → parse failure → fix → redeploy could happen multiple times per hour. Claude Code's ability to generate fix scripts, update definitions, and re-test in a single conversational turn eliminated the friction between iterations.

Cross-Domain Translation

The project required fluency in Python, XAML, JSON schema, REST APIs, OAuth 2.0, ERP domain modeling, and LLM prompt engineering — all simultaneously. Claude Code context-switched between these domains seamlessly within single sessions.

Challenges Where It Got Difficult

No Local Runtime for Validation

Unlike Python or JavaScript where code runs locally and errors are immediate, every Infor OS component lives in the cloud. Claude Code could generate a tool definition, but couldn't validate it without deploying to the GenAI platform and invoking it through the Chat API. The feedback loop was inherently slower than local development.

Proprietary Domain Knowledge Gap

Infor-specific concepts — IDO naming conventions (SLPOs not SLPos), SyteLine update format (Action=1 for insert), property names (Phone not VadPhone), site codes, logical IDs — aren't in any public training data. Every discovery required API exploration, trial-and-error, or reading internal documentation. Claude Code couldn't "just know" the platform.

LLM-to-LLM Indirection

The GenAI tools are interpreted by another LLM (Claude Sonnet on the Infor platform). The api_docs field is essentially a "prompt for the other AI." Formatting it so that the remote LLM consistently generated correct API calls was the hardest problem — a single blank line caused non-deterministic 404 failures. This meta-level debugging (AI debugging AI-generated prompts for another AI) had no established patterns.

XAML Without a Visual Designer

RPA Studio has a visual drag-and-drop designer. Claude Code generates raw XAML text. While the output was functional, complex layout adjustments (ViewStateData coordinates, activity sizing) were effectively blind edits. Some workflows needed minor visual cleanup in the designer after generation.

Platform API Instability

The GenAI Chat API changed its response format mid-project (JArray → JObject, "messages" → "items" property). IDP had V1/V2 incompatibilities. These platform-side changes required reactive fixes that couldn't be anticipated. Each change triggered a cascade of script and workflow updates.

Abandoned Paths Cost Context

The PO Receiving feature (session-scoped temp tables) and the V3/V4 experimental architectures (26 files) were dead ends. Claude Code faithfully built complete implementations before discovering they were infeasible. Domain expertise upfront could have avoided some of these, but the exploratory nature of the project made some dead ends inevitable.

Key Insights: AI-Assisted Development on Cloud Platforms

1

Persistent memory is essential

The CLAUDE.md + log.md convention turned Claude Code from a stateless tool into a knowledge-accumulating development partner. Each session built on the last. Without this, the same IDO casing bug would have been rediscovered repeatedly.

2

API specs are the source of truth

Feeding Swagger JSON files directly to Claude Code was more effective than any documentation. The AI could reason about endpoints, parameters, and response shapes directly from the spec, generating correct API calls faster than reading docs.

3

Test-via-API unlocks cloud development

The pattern of deploying tools/agents via script and then testing them through the Chat API created a workable feedback loop even without local runtimes. The test scripts became both verification and documentation.

4

Dead ends are part of the process

The 26-file V3/V4 experiment wasn't wasted — it proved that the V2 approach was correct and that the real issue was api_docs formatting, not architecture. AI's ability to rapidly prototype and discard is actually an advantage for exploratory work.

5

Domain knowledge must be taught, not assumed

Claude Code doesn't know Infor's IDO naming conventions or SyteLine's update format. But once taught (via CLAUDE.md or inline corrections), it applies that knowledge consistently. The initial investment in documenting platform quirks pays off exponentially.

6

Cross-domain fluency is the killer feature

No single human developer is equally fluent in Python, XAML, REST APIs, OAuth, JSON Schema, ERP modeling, and LLM prompt engineering. Claude Code's ability to operate across all these domains simultaneously — in a single conversation — is what made the project's velocity possible.

The Bottom Line

Claude Code was not just a code generator — it was the primary development environment for a project that touched 6 Infor OS services, produced 237+ files, and shipped a production V1.0 in approximately 8 working days. The biggest challenge was the gap between local development and cloud-native platforms: without local runtimes, every validation required a cloud round-trip. The biggest strength was velocity through iteration: Claude Code could generate, deploy, test, debug, fix, and redeploy faster than any traditional development workflow — especially across the multiple languages and technologies this project demanded.
