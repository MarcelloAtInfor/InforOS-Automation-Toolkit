The Development Process
Discovery-Driven Development on a Cloud ERP
The BOM Generator couldn't be planned upfront. SyteLine's BOM data model — the fact that BOMs are Standard Jobs, not direct item-material links — had to be discovered through IDO metadata queries. Each failed BOM attempt revealed new requirements (resource groups, schedule drivers, warehouse linkage) that expanded the tool count from 8 to 16. The development process was fundamentally exploratory: build, deploy, test against the live ERP, discover what's missing, add it, repeat.
1

Discover via Metadata

Query IDO metadata system IDOs (IdoCollections, IdoTables, IdoProperties) to discover writable fields, primary tables, and join relationships. Write discovery scripts. Map the domain model that doesn't exist in any public docs.

2

Build & Deploy Tools

Generate JSON tool specs with api_docs, anti-hallucination constraints, and responseInstructions. Validate via GAF_CLI. Publish to GenAI platform. Each tool = 1 IDO operation. No local runtime — everything runs in the cloud.

3

Test via BOM Creation

Send natural language prompts through the GenAI agent. Watch it call tools against live SyteLine. Verify results in ERP. Parse failures: 404s, silent write drops, hallucinated properties, IDO quirks. Each failure teaches.

4

Fix, Expand, Document

Fix the discovered issue. Add new tools if needed. Update agent workflow. Redeploy. Document the discovery in CLAUDE.md so it's never lost. The knowledge base compounds — each fix prevents the same class of error forever.

Development Timeline
Three Days of Discovery, Failure, and Recovery
The project compressed what would traditionally be weeks of ERP integration work into 3 days of high-velocity, AI-assisted exploration. Each day had a distinct character: Day 1 was foundational discovery and rapid tool building, Day 2 was dominated by failures that drove architectural pivots, and Day 3 brought the async breakthrough and full automation.
February 10, 2026 — Day 1
IDO Discovery & Phases 0–4 Discovery
Project initialized. Claude Code queried IDO metadata system to discover that SyteLine BOMs are Standard Jobs (Type='S'), not direct item-material links. Cataloged 88 work centers. Built and deployed 8 tools + 1 agent across 5 rapid phases in a single session.
The foundational discovery — that SLBomListing and SLMaterialListing are read-only views and the real write path goes through SLJobs → SLJobRoutes → SLJobmatls — came from querying IdoTables with TableType=3. This single metadata query shaped the entire project architecture.
February 10 (continued)
Phase 5 Testing: First Failures Hallucination
Agent hallucinated IDO properties ("Source" on SLItems), used invalid PMTCode "B" (only M/P/T valid), tried ProductCode "RM-100" (invalid — safe default is "FG-100"), and job auto-assign crashed with NextKeys_mst error. Anti-hallucination constraints added to all tool specs.
February 11, 2026 — Day 2
TB-10001 Toolbox: URL Hallucination 404s
The LLM on the Infor platform constructed completely malformed URLs — injecting tenant names, doubling service paths, using tool names as IDO endpoints. Every API call returned 404. Agent reported false success.
Root cause: the api_docs field used absolute URLs, and the GenAI framework auto-prepends servicePath. The LLM then further embellished the URL with hallucinated segments. Fix: all endpoints changed to relative paths + explicit URL construction rules added to every tool spec and the agent workflow. This was the project's "LLM-to-LLM meta-engineering" moment.
February 11 (continued)
GB-10000 Garden Bench: Tool Split Discovery Architecture
Two-step tools (GET then POST in one api_docs) only executed Step 1. The GenAI framework reads api_docs, makes one HTTP call, and the tool invocation ends. SetuprgidUpdate and ItemWhseUpdate never POSTed. BOM created but invisible in SyteLine UI.
This revealed a fundamental constraint of the GenAI tool framework: 1 tool = 1 HTTP call. The solution was splitting two-step operations into separate Load (GET) and Update (POST) tools, with the Load tool returning _ItemId that the Update tool requires. Tool count jumped from 8 to 14.
February 11 (continued)
Architectural Overhaul: Resource Groups + ItwhJob Pivot
IDO metadata investigation reversed a wrong conclusion about resource group writability. Discovered that SLJrtResourceGroups HAS data (IDO quirk: must specify properties param). Rgid is case-sensitive. ItwhJob must be set explicitly. Without it, BOMs are invisible in the Current Operations form.
The IDO metadata discovery technique (querying IdoCollections, IdoTables, IdoProperties) was elevated to a cross-project best practice after this session. A previous "0 records" conclusion about resource groups was caused by an IDO quirk: one IDO requires the properties param while another breaks if you include it. This kind of per-IDO behavioral variance can only be discovered empirically.
February 11 (continued)
BH-10000 Birdhouse: FIRST FULL SUCCESS Pass
After 4 failed attempts (TB-10001, GB-10000, SK-10000 partial, BOMTEST-001), the birdhouse BOM completed all 3 chunks. All 15 tools executed correctly. Resource groups populated, ItwhJob set, SchedDrv aligned. BOM fully visible in SyteLine UI.
February 11 (continued)
SK-10000 Skateboard: Resource Group Naming Failure Hard Error
Agent guessed Rgid from naming convention ("PG-300-LRG") but the actual value was "PG-300-MRG". Case-sensitive, atomic batch rollback killed all resource groups. Agent falsely reported "fully functional." New ResourceGroupLookup tool created. Error reclassification: RG failures = hard stop.
February 12, 2026 — Day 3
Async Chat Pattern: The 240s Breakthrough Discovery
After BH-T04000 "timed out" at 240.1s, the user checked Chat UI History and found the agent had actually completed successfully. The 240s limit is a chat_sync API gateway constraint, not a model execution limit. Switched to POST /chat (fire-and-forget) + polling. Eliminates the timeout entirely.
Two bugs were discovered during async implementation: (1) Session IDs must be dashless hex (uuid.uuid4().hex), not dashed UUIDs — the platform stores them dashless but silently accepts dashed ones, causing mismatches on retrieval. (2) Messages aren't stored until the agent finishes (~2-3 min) — 0 messages during polling is normal, not an error. Pagination is also 1-based (page=0 returns 422).
February 12 (continued)
Prompt Generator + JobNumberLookup Self-Serve
Created generate_prompt.py with 8 templates and dynamic BOM job number allocation via SLHighKeys API. Built JobNumberLookup_Tool_v1 (tool #16) so the agent can self-serve for ad-hoc users who don't specify job numbers. Confirmed BOM prefix auto-registration in SLHighKeys.
February 12 (continued)
BH-T03000: First Automated End-to-End Test 12/12
Full pipeline: generate_prompt.py → test_bom_agent.py → verify_bom.py. Birdhouse with 2 sub-assemblies, 3 standard jobs, 12 items. 5 chunks, 958.9s (~16 min). 12/12 verification checks passed. Multi-level BOM with RefType='J' sub-assemblies validated.
February 12 (continued)
Weighted Tool Budgets: LIGHT vs HEAVY Optimization
BH-T04000 chunk 4 hit 240s with 5 all-HEAVY calls averaging 48s each. Flat budget of 5 calls/chunk was unsafe for heavy operations. Introduced weighted classification: LIGHT-only chunks get 4 calls, HEAVY chunks get 3. Worst case: 3 × 50s + finish = 200s (40s margin).
February 12 (continued)
ST-10000 Stool: Ad-Hoc Self-Serve Validated 16/16
Prompt with no job numbers: "Create a BOM for a wooden stool." Agent called JobNumberLookup, allocated BOM0000805-808, reused 5 existing items, created 4 new ones. 8 chunks, ~16 min effective time (excluding 2 platform system errors that the agent recovered from). 16/16 verification checks passed.
Analysis
Strengths & Challenges
Building a 16-tool GenAI agent on a proprietary ERP platform with Claude Code surfaced distinct patterns of strength and difficulty. The most impactful strengths were in systematic discovery and iterative recovery; the hardest challenges involved invisible platform behaviors.

Strengths Where Claude Code Excelled

IDO Metadata Reverse Engineering

Claude Code treated SyteLine's metadata system IDOs (IdoCollections, IdoTables, IdoProperties) as queryable databases. By writing discovery scripts that systematically probed which tables were primary (writable) vs. joined (read-only), it mapped a BOM data model that exists in no public documentation. This technique became a cross-project best practice.

Failure-Driven Architecture

Each failed BOM produced a specific, actionable architectural fix: TB-10001's 404s led to URL construction rules, GB-10000's silent failures led to the tool-split pattern, SK-10000's naming guess led to the ResourceGroupLookup tool. Claude Code didn't just fix bugs — it converted each failure into a reusable constraint that prevented the entire class of error.

Anti-Hallucination Engineering

The project required writing "prompts for another LLM" (the GenAI platform's model interprets tool api_docs). Claude Code developed anti-hallucination patterns: explicit property lists ("ONLY use these exact properties"), URL construction rules ("endpoints MUST be relative"), error severity classification ("404 = HARD STOP"). These patterns were applied consistently across all 16 tools.

Progressive Complexity Management

The agent workflow grew from 3 steps to 9 steps, from 8 tools to 16, from static 3-chunk execution to dynamic weighted chunking. Claude Code managed this complexity growth by introducing abstractions only when needed: tool-call budgets, LIGHT/HEAVY classification, sentinel-based checkpoints, and a 5-point completion criteria.

Automated Test Pipeline Construction

Claude Code built a complete test-to-verify pipeline: unique prompt generation with dynamic job numbers from SLHighKeys, async multi-turn test execution that bypasses the 240s gateway timeout, and deep BOM verification that queries every IDO in the creation chain. The pipeline caught real issues (chunking budget, sentinel format) before they reached production prompts.

Challenges Where It Got Difficult

Silent Write Failures

SyteLine returns 200 OK for IDO updates even when the value is silently discarded because the column lives on a joined table, not the primary table. The API gives no error, no warning, and no indication that the write was ignored. The only way to detect this is to read back the record and check. This consumed multiple investigation cycles.

Per-IDO Behavioral Quirks

SLJrtResourceGroups requires the properties parameter to return data (omitting it gives 0 records). SLItemwhses is the exact opposite: including properties returns 0 records, omitting it works. IdoProperties returns 0 records for SLHighKeys but the IDO works fine. Each IDO had unique quirks discoverable only through trial-and-error.

LLM Non-Determinism in BOM Structure

The same birdhouse prompt produced 2 sub-assemblies in one run (BH-T03000) and 4 sub-assemblies in another (BH-T04000). More sub-assemblies meant more heavy tool calls, pushing chunk 4 past the 240s timeout. The fix required both deterministic prompt templates and weighted tool-call budgeting — addressing the problem from both sides.

Platform Transient Errors

The GenAI platform intermittently returns "I'm sorry, a system error has occurred" with no actionable detail. During the ST-10000 stool test, 2 consecutive chunks hit platform system errors. The agent recovered correctly on retry, but the errors added ~490s of wasted time and required the sentinel detection system to handle "UNKNOWN" states gracefully.

The 1-Tool-1-Call Constraint

The GenAI framework executes exactly one HTTP call per tool invocation. Tools that need to GET a record's _ItemId before POSTing an update cannot be a single tool — the GET fires but the POST never does. Discovering this required creating a tool, deploying it, testing it, observing the silent partial execution, and then splitting every two-step tool into separate Load + Update pairs.

Key Insights: Building GenAI Agents with Claude Code on Infor OS

1

Every failed BOM is an architecture lesson

The 4 failed BOMs before first success weren't wasted — each one exposed a gap (URL rules, tool-split pattern, resource group lookup, error classification) that, once fixed, permanently improved the system. The cost of failure was low (test data in a demo tenant); the knowledge gained was permanent.

2

IDO metadata is the Rosetta Stone

SyteLine's system metadata IDOs (IdoCollections, IdoTables, IdoProperties) are the only reliable way to determine which fields are writable. API documentation, property names, and even successful 200 responses can all be misleading. The metadata discovery pattern should be the first step in any SyteLine integration project.

3

Anti-hallucination is a first-class concern

When one LLM writes tool definitions that another LLM interprets, hallucination compounds. The agent LLM invents properties, URL segments, and IDO names that look plausible but don't exist. Every tool spec needs explicit "ONLY use these exact values" constraints. This is a design pattern, not an afterthought.

4

Phased validation is non-negotiable

On a cloud platform with no local runtime, the only way to know if something works is to deploy and test it in the live environment. Building all 16 tools before testing would have created an impossible debugging surface. One phase at a time, with user validation between each, was the only viable approach.

5

The knowledge base compounds exponentially

By Day 3, the CLAUDE.md file contained 400+ lines of hard-won platform knowledge: IDO quirks, valid property values, naming conventions, error patterns, architectural constraints. Every Claude Code session started by reading this file, avoiding rediscovery of issues that had already been solved. The documentation was not overhead — it was the primary development accelerator.

6

Async patterns unlock complex agent workflows

The switch from synchronous chat_sync (240s gateway timeout) to asynchronous chat + polling was transformative. It didn't just fix timeouts — it enabled weighted chunking, longer operations, and graceful recovery from platform errors. Complex agent workflows on the GenAI platform fundamentally require async execution.

The Bottom Line

Claude Code built a 16-tool GenAI agent that creates multi-level Bills of Material in Infor SyteLine from natural language prompts — a system touching 10 IDO collections, 88 work centers, and a 9-step creation dependency chain — in approximately 3 working days. The development was fundamentally discovery-driven: SyteLine's BOM data model had to be reverse-engineered through metadata queries, each failed BOM attempt revealed new architectural requirements, and the tool count doubled as hidden platform constraints surfaced. The biggest challenge was invisible platform behavior — silent write failures, per-IDO quirks, and LLM-to-LLM hallucination compounding. The biggest strength was systematic failure recovery: Claude Code converted every failed BOM into reusable constraints, anti-hallucination patterns, and documented knowledge that prevented the same class of error from recurring. The result is not just a working agent, but a comprehensive knowledge base and automated test pipeline that makes the system maintainable and extensible.
